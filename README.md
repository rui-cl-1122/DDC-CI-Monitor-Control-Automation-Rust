# DDC-CI Monitor Control Automation (Rust Rewrite)

## このリポジトリについて

このリポジトリは、Windows 上で運用していた Python 製のモニタ制御プロトタイプをもとに、  
責務分離と境界設計を整理し直すために進めている Rust 実装です。

&nbsp;

## この実装の目的

Python版では、実運用を通じて必要な振る舞いを検証することを優先しました。  
その結果、機能としては成立した一方で、以下の課題が明確になりました。

- OS依存処理と制御ロジックの境界が曖昧になりやすい  
- モニタ識別が環境差分の影響を受けやすい  
- テストや差し替えを前提とした構造になっていない  
- 機能追加に伴い責務の分離が難しくなる  

本実装ではこれらを踏まえ、  
制御ロジック・外部依存・データ表現の境界を整理し直すことを目的としています。

&nbsp;

## 現在の実装範囲

現時点ではまだ初期段階であり、完成したアプリケーションを提供する段階ではありません。  
再設計の起点となる最小スライスの実装に焦点を当てています。

現在実装している主な内容は以下です。

- DDC/CI を通じたモニタ情報の取得  
- EDID を通じたモニタ識別情報の取得  
- DDC と EDID を統合したモニタ ID の組み立て  
- 取得したモニタ一覧の最小表示  

つまり現在は、後続の制御機能を安定して構築するための前提として、  
モニタをどう識別し、どう外部情報を取り込むかに焦点を当てています。

&nbsp;

## 設計方針

### 1. OS依存処理の分離
Windows API や DDC/CI などの外部依存は特定の層に集約し、  
アプリケーションロジックから直接参照されない構造を目指します。

### 2. ユースケース中心の構成
アプリケーション層は「何をするか」に集中し、  
外部とのやり取りは明確なインターフェースを通じて扱います。

### 3. 機能単位での責務整理
入力取得、状態解決、実行処理といった役割を分離し、  
それぞれが独立して変更可能な構造にします。

### 4. モニタ識別の安定化
DDC/CI に加えて EDID 情報を組み合わせることで、  
環境差分に影響されにくい識別方法を構築します。

### 5. データ表現の整理
Python版では VCP コードをそのまま扱っていましたが、  
本実装では意味を持つ単位として扱えるように整理し、  
既知の項目はラベルベースで扱えるようにすることを検討しています。

&nbsp;

## 現在意識しているレイヤ構成

現時点では、概ね以下のような責務分離を意識しています。

### Application
- ユースケース  
- ポート定義  
- アプリケーションの中心となる振る舞い  

### Adapter
- Application が扱いやすい形への接続  
- ユースケースごとの入出力変換  

### Infra
- DDC/CI  
- EDID  
- WinAPI  
- OS や外部デバイスとの具体的な接続  

### Presentation
- GUI / 表示  
- ユーザーとの接点  

この構成はまだ調整中であり、  
特に Adapter と Infra の責務境界については継続的に見直しています。

&nbsp;

## 現在の設計上の関心

このリポジトリでは、現時点で主に以下を検討しています。

- OS依存をどこまで Infra に閉じ込められるか  
- Adapter をどの粒度で分けるべきか  
- 後続のユースケース追加に耐える構成になっているか  
- Windows専用実装で始めつつ、将来の拡張余地をどう確保するか
